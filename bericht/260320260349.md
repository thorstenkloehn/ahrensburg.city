# Sicherheitsbericht - MeinCMS (26.03.2026)

Dieser Bericht fasst die aktuellen Sicherheitsanalysen, identifizierte Schwachstellen und potenzielle Bugs im System zusammen.

## Zusammenfassung
Das System weist eine solide Grundsicherheit auf (gehärtete Password-Policy, globale Anti-Forgery-Filter, HSTS, CSP). Es wurden jedoch kritische Schwachstellen in der Mandanten-Isolierung und moderate Schwachstellen in der CSP-Konfiguration identifiziert.

## Produktionsreife (Kann das System auf einem Server laufen?)
**Eingeschränkt ja, aber mit dringendem Handlungsbedarf.**
Das System ist technisch funktionsfähig und verfügt über wichtige Sicherheitsmechanismen (HSTS, Anti-Forgery, Sanitisierung). Ein Einsatz auf einem produktiven Server ist möglich, **allerdings nur unter der Bedingung, dass Punkt 1.1 (Mandanten-Spoofing) umgehend behoben wird**. Solange diese Schwachstelle besteht, könnten Mandantendaten durch manipulierte Host-Header kompromittiert werden. 

Zusätzlich sollte für einen echten Produktionsbetrieb sichergestellt sein, dass der Server (z.B. Nginx/Apache als Reverse Proxy) eingehende Anfragen streng auf zulässige Hostnamen filtert, um die Schwachstelle im `TenantService` netzwerkseitig zu mitigieren.

---

## 1. Kritische & Hohe Priorität

### 1.1 Mandanten-Spoofing (Hostname-Injection)
*   **Datei:** `mvc/Services/TenantService.cs`
*   **Schwachstelle:** Die Prüfung `host.Contains(e.Key.ToLower())` ist unzureichend.
*   **Risiko:** Angreifer können über fremde Domains (z. B. `doc.localhost.evil.com`) auf Mandanten-Daten zugreifen, falls der Server diese Anfragen entgegennimmt. Dies bricht die Datenisolation der Multi-Tenancy.
*   **Empfehlung:** Änderung von `.Contains()` auf `.Equals()` oder eine strikte Suffix-Prüfung (z. B. `host == key || host.EndsWith("." + key)`).

### 1.2 Unvollständige Mandanten-Isolierung in Admin-Tools
*   **Datei:** `UserAdmin/Program.cs`
*   **Bug:** Der Import/Export-Prozess nutzt keine globalen Query-Filter oder `IgnoreQueryFilters()`.
*   **Risiko:** Wenn Daten für einen Mandanten (z. B. `doc`) importiert werden, das Tool aber im Kontext von `main` läuft, erkennt es existierende Artikel nicht und versucht, Dubletten anzulegen, was zu Datenbankfehlern (Unique Constraint Violation) führt.
*   **Status:** In `backup/Program.cs` bereits behoben, in `UserAdmin/Program.cs` noch offen.

## 2. Mittlere Priorität

### 2.1 CSP: 'unsafe-inline' in Script/Style-Sources
*   **Datei:** `mvc/Program.cs`
*   **Schwachstelle:** Die Content Security Policy erlaubt `'unsafe-inline'`.
*   **Risiko:** Erhöht das Risiko von Cross-Site Scripting (XSS), falls die Eingabe-Bereinigung (`HtmlSanitizer`) umgangen werden kann.
*   **Empfehlung:** Nutzung von Nonces oder Hashes für Inline-Scripte und strikte Trennung von CSS/JS in Dateien.

### 2.2 Fehlende Bind-Einschränkungen bei Modellen
*   **Dateien:** `mvc/Models/WikiArtikel.cs`, `WikiArtikelVersion.cs`
*   **Schwachstelle:** `TenantId` ist nicht vor Model-Binding geschützt (`[BindNever]`).
*   **Risiko:** Auch wenn der `PageController` aktuell nur explizite Parameter nutzt, könnte bei zukünftigen Erweiterungen (z. B. Nutzung von `[Bind]`) ein Angreifer versuchen, die `TenantId` über einen POST-Parameter zu manipulieren.
*   **Empfehlung:** Explizites Hinzufügen von `[BindNever]` oder Nutzung von DTOs.

---

## 3. Niedrige Priorität & Bugs

### 3.1 Unix Socket Berechtigung
*   **Datei:** `mvc/Program.cs`
*   **Beobachtung:** Der Socket unter `/run/meincms.sock` wird beim Start gelöscht und neu erstellt.
*   **Risiko:** Falls der Prozess nicht mit ausreichenden Rechten läuft oder der Pfad unsicher ist, kann der Start fehlschlagen.
*   **Empfehlung:** Sicherstellen, dass das Verzeichnis `/run/` dem Service-User gehört oder ein Unterverzeichnis genutzt wird.

### 3.2 HTML-Sanitizer Konfiguration
*   **Datei:** `mvc/Services/PageService.cs`
*   **Beobachtung:** Der `HtmlSanitizer` wird mit Standard-Einstellungen genutzt.
*   **Empfehlung:** Prüfung, ob bestimmte Tags (z. B. `<iframe?`, `<embed>`) explizit erlaubt oder verboten werden müssen, um Phishing oder Clickjacking innerhalb der Wiki-Inhalte zu verhindern.

---

## Erledigte Punkte (Post-Audit Fixes)
- [x] **Slug-Validierung:** Erweitert um Unicode-Unterstützung und URL-Sonderzeichen (`%`, `.`).
- [x] **Tenant-Overwrite Fix:** `ApplicationDbContext` setzt `TenantId` nur noch, wenn diese leer ist, was administrative Korrekturen ermöglicht.
- [x] **HSTS Preload:** Korrekt konfiguriert für 365 Tage.
- [x] **Password Policy:** Erhöht auf 12 Zeichen mit Komplexitätsanforderungen.
