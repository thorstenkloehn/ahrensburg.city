# Sicherheits-Audit & Projektbericht - 26.03.2026

## 1. Statusbericht (Aktuelle Änderungen)

In der heutigen Sitzung wurden kritische Verbesserungen an der Datenintegrität und der Backup-Infrastruktur vorgenommen:

- **Backup-Tool Optimierung**: 
    - Unterstützung für **YAML** als kompaktes Austauschformat eingefügt.
    - Reduzierung der Dateigröße um ca. 60-80% durch Ausschluss des redundanten `HtmlInhalt` beim Export (wird beim Import automatisch regeneriert).
    - Unterstützung für beliebige Dateinamen beim Export/Import.
- **Datenbereinigung (Multi-Tenancy)**:
    - Es wurde eine Schwachstelle identifiziert, bei der Artikel mit leerer `TenantId` in der Datenbank existierten. Diese wurden durch den globalen Query-Filter (`TenantId == "main"`) im Web-Interface ausgeblendet.
    - Das Backup-Tool wurde um eine **Normalisierungs-Logik** erweitert, die leere `TenantId`-Felder beim Export/Import automatisch zu `"main"` korrigiert.
    - Ein vollständiger Re-Import hat die Datenbank konsolidiert.
- **Branding**:
    - Der Titel des Hauptmandanten wurde von "Ahrensburg Wiki" auf **"ahrensburg.city"** geändert.

## 2. Schwachstellenanalyse (Vulnerabilities & Bugs)

### Bekannte Schwachstellen
1.  **EF Core Versionskonflikt**: Beim Build treten Warnungen (MSB3277) bezüglich `Microsoft.EntityFrameworkCore.Relational` (10.0.1 vs 10.0.2) auf. Dies ist aktuell unkritisch, sollte aber durch eine Konsolidierung der NuGet-Pakete in der Solution behoben werden.
2.  **Manuelle User-Verwaltung**: Die Abhängigkeit vom `UserAdmin`-Tool ist sicher, aber unflexibel für skalierende Teams. Es fehlt eine (mandantenspezifische) Admin-UI für die Benutzerverwaltung.
3.  **Fehlendes Rate-Limiting**: Das Wiki-Routing ist offen. Ein Angreifer könnte durch massenhafte Anfragen auf nicht existierende Slugs die Datenbanklast erhöhen (Denial of Service).

### Identifizierte Bugs
- [Behoben] **Verschwindende Artikel**: Artikel ohne explizite `TenantId` wurden durch den Query-Filter fälschlicherweise versteckt.
- [Offen] **Slug-Kollisionen bei Umbenennung**: Wenn ein Slug geändert wird, bleibt der alte Slug nicht als Redirect erhalten, was zu 404-Fehlern bei externen Links führt.

## 3. Schutz durch snyk.io

Um die Anwendung weiter abzusichern, wird die Integration von **Snyk** dringend empfohlen:

1.  **Snyk Open Source (SCA)**:
    - Überprüft alle NuGet-Abhängigkeiten (z.B. `Markdig`, `HtmlSanitizer`) auf bekannte Sicherheitslücken (CVEs).
    - *Vorschlag*: Integration in eine GitHub Action oder lokale Ausführung via `snyk test`.
2.  **Snyk Code (SAST)**:
    - Scannt den C#-Code auf unsichere Muster (z.B. SQL-Injection, unsichere Zufallszahlen, fehlende CSRF-Token).
    - Da wir `HtmlSanitizer` nutzen, prüft Snyk, ob dieser an allen relevanten Stellen (Eingabe/Ausgabe) korrekt angewendet wird.
3.  **Snyk Container**:
    - Falls das Projekt in Docker betrieben wird, scannt Snyk das Basis-Image (z.B. `mcr.microsoft.com/dotnet/aspnet:10.0`) auf Betriebssystem-Schwachstellen.
4.  **Snyk IaC**:
    - Scannt Konfigurationsdateien (wie Kubernetes-Manifeste oder Terraform) auf Fehlkonfigurationen.

## 4. Vorschläge & Weiterführende Fragen

### Strategische Vorschläge
- **Redirect-Manager**: Implementierung einer Tabelle für Slugs, die umbenannt wurden, um SEO-Ranking zu erhalten.
- **Zwei-Faktor-Authentifizierung (2FA)**: Da das Wiki sensible Informationen enthalten kann, sollte für Administratoren 2FA (TOTP) im Identity-System aktiviert werden.
- **Media-Management**: Aktuell gibt es keinen Bild-Upload. Ein sicherer Proxy für Bilder oder eine Integration von S3/Azure Blob Storage mit Malware-Scan wäre ein logischer nächster Schritt.

### Fragen an die Architektur
- Sollen Mandanten gegenseitig Artikel "teilen" können (Shared Content)?
- Ist eine Volltextsuche (z.B. via PostgreSQL GIN-Index oder Lucene) geplant, da das Wiki wächst?

## 5. Fazit
Das System ist technologisch auf dem modernsten Stand (.NET 10). Die heutige Bereinigung der Mandanten-Daten hat die Zuverlässigkeit erheblich gesteigert. Der Fokus sollte nun auf **automatisierter Security-Pipeline (Snyk)** und **Usability für Administratoren** liegen.

## 6. Empfehlung zur Veröffentlichung (Produktionsreife)

**Frage:** Kann die Seite auf einem Produktionsserver im Internet veröffentlicht werden?

**Antwort:** **Eingeschränktes JA (Soft Launch bereit).**

Das System ist stabil und sicherheitstechnisch gehärtet. Für einen sicheren Betrieb im offenen Internet sollten jedoch folgende Begleitmaßnahmen auf Server-Ebene (Nginx/OS) sichergestellt sein:

1.  **Nginx Rate-Limiting**: Da die App intern noch keinen DoS-Schutz für das Wiki-Routing hat, muss Nginx Anfragen pro Sekunde pro IP limitieren (`limit_req`).
2.  **SSL/TLS**: Ein Betrieb ohne HTTPS (Let's Encrypt) ist aufgrund der Identity-Anmeldung (Passwörter) absolut untersagt.
3.  **Monitoring**: Da wir Unix-Sockets nutzen, sollte ein Watchdog (z.B. Systemd) den Dienst bei Fehlern automatisch neu starten.
4.  **Snyk-Scan**: Vor dem finalen "großen" Release sollte einmalig ein vollständiger Snyk-Scan der NuGet-Pakete durchgeführt werden, um 0-Day-Lücken in Drittanbieter-Bibliotheken auszuschließen.

**Fazit:** Wenn Nginx korrekt konfiguriert ist, steht einem Start unter **ahrensburg.city** nichts im Wege.
