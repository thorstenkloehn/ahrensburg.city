# Nachtragsbericht Multi-Tenancy Implementierung - MeinCMS (24.03.2026)

Dieser Bericht dokumentiert die Einführung der Mandantenfähigkeit (Multi-Tenancy) für den Parallelbetrieb von `ahrensburg.city` und `doc.ahrensburg.city`.

## 1. Zusammenfassung der Änderungen
- **Architektur-Upgrade:** Das System wurde von einer Single-Tenant-Architektur auf eine Multi-Tenant-Architektur umgestellt.
- **Mandantenerkennung:** Ein neuer `TenantService` ermittelt den aktiven Mandanten (`main` vs. `doc`) basierend auf dem Hostnamen des Eingehenden Requests.
- **Datenbank-Migration:** Alle Kern-Tabellen (`WikiArtikel`, `WikiArtikelVersion`) wurden um eine `TenantId`-Spalte erweitert.

## 2. Sicherheits- und Schwachstellen-Analyse

### 2.1 Datenleck-Prävention (Cross-Tenant Data Leakage) - *Status: Sicher*
- **Problem:** In einer geteilten Datenbank besteht das Risiko, dass Mandant A versehentlich Daten von Mandant B sieht oder manipuliert.
- **Untersuchung:** Alle LINQ-Abfragen müssten theoretisch manuell um ein `Where(a => a.TenantId == ...)` ergänzt werden, was fehleranfällig ist (menschliches Versagen).
- **Lösung:** Implementierung von **Global Query Filters** in Entity Framework Core. Der Filter wird automatisch auf alle Abfragen angewendet, was ein vergessenes Filtern im Code technisch unmöglich macht.
- **Status:** Das Risiko für Cross-Tenant Data Leakage wurde durch diese architektonische Maßnahme minimiert.

### 2.2 Slug-Kollisionen (Logical Bug) - *Status: Behoben*
- **Problem:** Zuvor war der URL-Slug über die gesamte Tabelle hinweg eindeutig. Es wäre unmöglich gewesen, auf beiden Domains eine Seite namens "Impressum" zu führen.
- **Lösung:** Der eindeutige Index wurde von `Slug` auf ein Tupel `(TenantId, Slug)` geändert. 
- **Status:** Jeder Mandant verfügt nun über einen eigenen, isolierten Namensraum für seine Wiki-Seiten.

### 2.3 Automatisches Speichern (Data Integrity) - *Status: Sicher*
- **Problem:** Manuelles Setzen der `TenantId` beim Speichern ist fehleranfällig.
- **Lösung:** Überschreiben der `SaveChangesAsync`-Methode im `ApplicationDbContext`. Die `TenantId` wird nun beim `Insert`-Vorgang automatisch aus dem aktuellen Kontext (via `ITenantService`) injiziert.

## 3. Identitätsmanagement ( Identity) - *Status: Unverändert*
- **Status:** Die Benutzerkonten (Administratoren) sind weiterhin global in der `AspNetUsers`-Tabelle gespeichert.
- **Implikation:** Ein Administrator hat Zugriff auf alle Mandanten. Dies ist für das aktuelle Szenario (Zentralverwaltung beider Seiten) so vorgesehen, sollte aber bei einer Ausweitung auf externe Mandanten überdacht werden.

## 4. Fazit und Empfehlung
Die Implementierung der Mandantenfähigkeit wurde unter Berücksichtigung moderner Sicherheitsstandards (Global Query Filters) durchgeführt. Die Trennung der Inhalte ist auf Datenbankebene konsistent und gegen menschliche Fehler im Controller-Code abgesichert.

**Ergebnis:** Die Erweiterung ist stabil und sicher für den Betrieb beider Domains auf einer Instanz.
